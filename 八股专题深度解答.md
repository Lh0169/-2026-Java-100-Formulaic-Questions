## 一、MyBatis 专题

### Mybatis专题-01：什么是 MyBatis？

MyBatis 是一款优秀的**半自动 ORM（Object Relational Mapping）框架**，它支持自定义 SQL、存储过程以及高级映射。MyBatis 免除了几乎所有的 JDBC 代码以及设置参数和获取结果集的工作。

**核心定位**：介于全自动化 ORM（如 Hibernate）和纯 JDBC 之间的中间层，通过 XML 或注解方式将 Java 对象与 SQL 语句映射，让开发者专注于 SQL 本身。

**核心优势**：SQL 与代码分离便于优化；支持动态 SQL；提供一级/二级缓存；轻量级学习成本低。

---

### Mybatis专题-02：MyBatis 的优缺点？

**优点**：SQL 可控便于 DBA 优化；解耦（SQL 在 XML 中）；动态 SQL 灵活；映射灵活支持复杂关联；内置缓存；插件扩展；与 Spring 无缝集成。

**缺点**：SQL 编写工作量大；数据库移植性差；缺少全自动 ORM 便利；缓存使用需谨慎；XML 调试困难。

---

### Mybatis专题-03：MyBatis 与 JDBC、Hibernate 的核心区别？

| 对比项 | JDBC | MyBatis | Hibernate |
|--------|------|---------|-----------|
| 代码量 | 大量样板代码 | 几乎零样板代码 | 极少代码 |
| SQL 管理 | 硬编码 | XML/注解分离 | 自动生成 |
| 结果映射 | 手动遍历 ResultSet | 自动映射 | 全自动映射 |
| 缓存 | 无 | 一级+二级 | 一级+二级+查询缓存 |
| 动态 SQL | 手动拼接（易注入） | 标签式（安全） | HQL/Criteria |
| 适用场景 | 极致性能 | 复杂查询 | 快速开发 |

---

### Mybatis专题-04：为什么说 MyBatis 是半自动 ORM？

**半自动含义**：MyBatis 只完成对象-关系映射的部分工作，SQL 仍需开发者手动编写。

**体现**：SQL 需手写（非自动生成）；ResultMap 需配置；关联关系需手动处理（`<association>`/`<collection>`）。

**对比 Hibernate**：Hibernate 自动生成 SQL，开发者几乎不接触 SQL；MyBatis 开发者写 SQL，框架只负责执行和映射。

---

### Mybatis专题-05：MyBatis 适用场景？

复杂查询场景（多表关联、子查询）；遗留系统改造；性能敏感型应用；多数据库环境；团队 SQL 能力强；Spring Boot 集成。

**不适用**：快速原型开发；频繁切换数据库；团队缺乏 SQL 能力。

---

### Mybatis专题-06：MyBatis 核心组件有哪些？

**接口层**：SqlSession（核心会话）、Mapper 接口（动态代理）

**核心处理层**：Executor（SQL 执行器）、StatementHandler（Statement 操作）、ParameterHandler（参数处理）、ResultSetHandler（结果映射）、SqlSource/BoundSql（SQL 解析）、MappedStatement（SQL 配置）

**基础支撑层**：Configuration（全局配置）、Transaction（事务）、DataSource（数据源）、TypeHandler（类型转换）、Cache（缓存）、Plugin（插件）

---

### Mybatis专题-07：SqlSession 的生命周期？

**创建**：`SqlSessionFactory.openSession()` 创建，初始化 Executor、Transaction。
**使用**：执行 SQL 操作、获取 Mapper 代理、管理事务。
**关闭**：必须 `close()` 释放资源，一级缓存失效。

**最佳实践**：线程不安全，每个线程独立实例；用完即关（try-with-resources）；Spring 集成时由 `SqlSessionTemplate` 管理。

---

### Mybatis专题-08：Mapper 接口为什么不需要实现类？

**核心原理：JDK 动态代理**。MyBatis 通过 `MapperProxyFactory` 创建代理对象，`MapperProxy` 实现 `InvocationHandler` 拦截方法调用。

**流程**：调用接口方法 → 代理拦截 → 根据 `namespace.methodName` 找到 MappedStatement → 通过 SqlSession 执行 SQL。

**原因**：接口方法签名即 SQL ID；动态代理拦截转发；解耦 SQL 与 Java 代码。

---

### Mybatis专题-09：MyBatis 工作流程？

**启动初始化**：读取配置 → 构建 SqlSessionFactoryBuilder → 解析 XML 创建 Configuration → 解析 mappers 生成 MappedStatement → 创建 SqlSessionFactory。

**SQL 执行**：创建 SqlSession → 获取 Mapper 代理 → 调用方法触发代理 → 找到 MappedStatement → Executor 执行（先查一级缓存 → 二级缓存 → DB）→ StatementHandler 创建 Statement → ParameterHandler 设置参数 → JDBC 执行 → ResultSetHandler 映射 → 存入缓存 → 返回。

---

### Mybatis专题-10：MyBatis 有几种 Executor 执行器？

| 执行器 | 特点 | 适用场景 |
|--------|------|----------|
| **SimpleExecutor** | 每次创建新 Statement | 默认，普通场景 |
| **ReuseExecutor** | 复用 Statement | 同 SQL 多次执行 |
| **BatchExecutor** | 批量执行统一提交 | 批量插入/更新 |

**CachingExecutor**：装饰器模式，包装 BaseExecutor 实现二级缓存。

---

### Mybatis专题-11：TypeHandler 作用？

实现 **Java 类型** 与 **JDBC 类型** 之间的转换。

**内置**：StringTypeHandler、IntegerTypeHandler、DateTypeHandler、EnumTypeHandler 等。

**自定义**：继承 `BaseTypeHandler`，处理特殊类型（如 JSON ↔ String）。

---

### Mybatis专题-12：如何配置 Mapper？

1. **XML 映射文件**：`<mapper resource="..."/>`
2. **包扫描**：`<package name="com.example.mapper"/>`（推荐）
3. **注解方式**：`@Select`、`@Insert` 等（无需 XML）
4. **Spring Boot**：`mybatis.mapper-locations` + `@MapperScan`

---

### Mybatis专题-13：MyBatis 如何集成 Spring？

**核心依赖**：`mybatis-spring`

**核心配置**：`SqlSessionFactoryBean`（创建工厂）、`SqlSessionTemplate`（线程安全会话）、`@MapperScan`（扫描 Mapper）、`DataSourceTransactionManager`（事务管理）。

### Mybatis专题-14：Spring Boot 整合 MyBatis 核心步骤？

2. 配置数据源和 MyBatis（`application.yml`）

### Mybatis专题-26：一级缓存是什么？

**会话级缓存**（SqlSession 级别），默认开启，HashMap 存储。

**失效**：执行增删改、手动 `clearCache()`、SqlSession 关闭、`flushCache="true"`。

---

### Mybatis专题-27：二级缓存是什么？

**Mapper 级别缓存**，跨 SqlSession 共享，默认关闭。

**开启**：全局 `cacheEnabled=true` + Mapper `<cache/>` + 实体类实现 `Serializable`。

**淘汰策略**：LRU（默认）、FIFO、SOFT、WEAK。

---

### Mybatis专题-28：二级缓存失效场景？

- 同一 namespace 执行增删改
- 未提交事务（未 commit 不刷新到二级缓存）
- 跨 namespace 操作（相互独立）
- `flushCache="true"` 或 `useCache="false"`
- 分布式环境（建议用 Redis 替代）

---

### Mybatis专题-29：延迟加载（懒加载）如何配置？

```xml
<setting name="lazyLoadingEnabled" value="true"/>
<setting name="aggressiveLazyLoading" value="false"/>
```

Mapper 中：`fetchType="lazy"`（`fetchType="eager"` 立即加载）

---

### Mybatis专题-30：延迟加载原理？

**CGLIB 动态代理**。关联属性初始为代理对象，访问时触发拦截器执行关联查询 SQL，加载真实数据。

### Redis专题-02：Redis 的适用场景

缓存、分布式 Session、分布式锁、计数器、限流、消息队列、排行榜、社交网络、地理位置、实时系统、数据过期。

---

### Redis专题-03：Redis 的优缺点

**优点**：高性能（10万+ QPS）、数据类型丰富、持久化（RDB/AOF）、高可用（主从/哨兵/集群）、原子性、Pub/Sub、Lua 脚本、轻量级。

**缺点**：内存限制、持久化性能损耗、单线程瓶颈（CPU 密集型）、事务弱（无回滚）、数据丢失风险（异步复制）、冷数据问题、集群复杂。

---

### Redis专题-04：Redis 与其他数据库对比

| 特性 | Redis | Memcached | MongoDB | MySQL |
|------|-------|-----------|---------|-------|
| 数据模型 | 键值对 | 纯键值对 | 文档型 | 关系型 |
| 持久化 | RDB+AOF | 无 | 支持 | 支持 |
| 查询 | 键查询 | 键查询 | 丰富查询 | SQL |
| 事务 | 弱 | 无 | 多文档 ACID | 强 ACID |
| 适用 | 缓存/实时 | 纯缓存 | 文档存储 | 事务型 |

---

### Redis专题-05：Redis 是单线程还是多线程？为什么设计为单线程？

**Redis 6.0 前**：单线程（网络 IO + 命令执行）
**Redis 6.0+**：多线程 IO（网络读写），命令执行仍为单线程。

**单线程原因**：避免上下文切换；避免锁竞争；内存操作快（瓶颈在网络 IO）；IO 多路复用（epoll/kqueue）。

**多线程 IO**：`io-threads 4`，仅处理网络 IO，命令执行单线程保证原子性。

---

### Redis专题-06：Redis 如何实现持久化？RDB 和 AOF 的区别是什么？

**RDB（快照）**：定时 fork 子进程将内存数据保存为二进制文件。优点：文件紧凑、恢复快；缺点：可能丢失最后一次快照后数据。

**AOF（日志）**：记录所有写命令，重启时重放。优点：数据安全（最多丢 1 秒）；缺点：文件大、恢复慢。

**混合持久化（Redis 4.0+）**：AOF 重写时先写 RDB 全量，再追加增量 AOF。

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件体积 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全 | 可能丢较多 | 最多丢 1 秒 |
| 性能影响 | fork 阻塞 | 刷盘影响写 |

---

### Redis专题-07：内存淘汰策略有哪些？（至少列出 4 种）

| 策略 | 说明 |
|------|------|
| noeviction | 不淘汰，返回错误 |
| allkeys-lru | 所有 key 中淘汰最近最少使用 |
| volatile-lru | 仅淘汰有过期时间的 key 中最近最少使用 |
| allkeys-random | 所有 key 随机淘汰 |
| volatile-random | 仅淘汰有过期时间的 key 随机淘汰 |
| allkeys-lfu | 所有 key 中淘汰使用频率最少（Redis 4.0+） |
| volatile-lfu | 仅淘汰有过期时间的 key 中使用频率最少 |
| volatile-ttl | 淘汰 TTL 最短的 |

---

### Redis专题-08：Redis 的 Hash 数据类型底层结构是什么？

**ziplist（压缩列表）**：字段少且短（<512 个，<64 字节）时使用，连续内存节省空间。

**hashtable（哈希表）**：字段多或长时使用，数组+链表，O(1) 查找。

**冲突解决**：链地址法。
**扩容**：负载因子 >= 1（或 >= 5 有后台持久化时）触发渐进式 rehash。

---

### Redis专题-09：Redis 的过期策略有哪些？

**惰性删除**：访问时检查过期，过期则删除。优点：不耗 CPU；缺点：过期 key 不被访问则占内存。

**定期删除**：每 100ms 随机抽 20 个 key 检查，过期比例 > 25% 继续循环。平衡 CPU 和内存。

**内存淘汰**：兜底机制。

---

### Redis专题-10：Redis 的持久化机制中，AOF 重写的过程是怎样的？

1. 触发重写（手动或自动）
2. fork 子进程（Copy-On-Write）
3. 子进程遍历内存生成新 AOF（只含最终状态）
4. 主进程继续处理命令，新命令写入 AOF 重写缓冲区
5. 子进程完成后，主进程将缓冲区数据追加到新 AOF
6. 原子替换旧 AOF

**目的**：压缩体积，去除中间状态命令。

---

### Redis专题-11：什么是缓存雪崩？如何解决？

**定义**：大量热点 key 同时过期或 Redis 集群宕机，请求全部穿透到 DB。

**解决**：
1. 过期时间错开（基础时间 + 随机值）
2. Redis 高可用（主从 + 哨兵 + 集群）
3. 缓存降级/熔断（返回默认值）
4. 多级缓存（本地缓存 → Redis → DB）
5. 缓存预热

---

### Redis专题-12：什么是缓存穿透？如何解决？

**定义**：请求的 key 在缓存和 DB 中都不存在，大量请求穿透到 DB。

**解决**：
1. 参数校验（拦截非法参数）
2. 空值缓存（缓存 null，设置短过期时间）
3. 布隆过滤器（预先加载有效 key，不存在直接返回）

---

### Redis专题-13：什么是缓存击穿？如何解决？

**定义**：热点 key 过期，大量并发请求同时访问，全部穿透到 DB。

**解决**：
1. 互斥锁（分布式锁，只允许一个线程查 DB）
2. 热点 key 永不过期（逻辑过期，后台异步更新）
3. 缓存预热

---

### Redis专题-14：什么是脑裂问题？如何避免？

**定义**：网络分区导致原主节点与哨兵失联，但仍在接收写请求；哨兵选举出新主节点，存在两个主节点。

**解决**：
1. `min-slaves-to-write 1` + `min-slaves-max-lag 10`（至少 1 个从节点连接才允许写）
2. 半同步复制（主节点写需从节点确认）
3. 合理配置哨兵参数（`down-after-milliseconds`）

---

### Redis专题-15：Redis 的热 Key 和大 Key 问题如何发现与解决？

**热 Key**：单个 key 被大量访问。
- 发现：`redis-cli --hotkeys`、监控 `keyspace_hits`
- 解决：本地缓存（Caffeine）、Key 拆分（hash 分片）、读写分离

**大 Key**：value 过大（Hash 百万字段、String 几十 MB）。
- 发现：`redis-cli --bigkeys`、`DEBUG OBJECT key`
- 解决：拆分（按 ID 取模）、压缩（Snappy/LZ4）、异步删除（`UNLINK`）、定期清理

---

### Redis专题-16：如何保证数据库与 Redis 的数据一致性？

**Cache Aside（最常用）**：
- 读：查缓存 → 命中返回；未命中查 DB → 写入缓存 → 返回
- 写：更新 DB → **删除缓存**（非更新缓存）

**延迟双删**：更新 DB 后，延迟一段时间再次删除缓存，防止并发读导致旧数据回填。

---

### Redis专题-17：什么是延迟双删策略？如何实现？

**问题**：删除缓存后、DB 更新完成前，并发读查到旧数据并写入缓存。

**流程**：删除缓存 → 更新 DB → 休眠 500ms（> 主从同步延迟）→ 再次删除缓存。

```java
redisTemplate.delete(key);
userMapper.update(user);
// 异步延迟双删
CompletableFuture.runAsync(() -> {
    Thread.sleep(500);
    redisTemplate.delete(key);
});
```

---

### Redis专题-18：Redis 的 Hash 冲突如何解决？扩容机制是怎样的？

**冲突解决**：链地址法，冲突元素组成链表。

**扩容**：负载因子 >= 1（或 >= 5 有后台持久化时）触发。
- 渐进式 rehash：创建新表，每次增删改查迁移一个桶，查询同时查两个表。

### Redis专题-19：跳跃表（SkipList）的结构和优势是什么？

**结构**：多层有序链表，底层包含所有元素，上层是快速通道。每个节点随机决定层数（1/2 概率晋升）。

**优势**：实现简单（比红黑树代码少）；范围查询高效 O(log N)；并发友好（局部指针修改）；性能稳定。

**应用**：ZSet 底层实现（元素多时用 skiplist）。

1. **用户签到**：一年 365 bit ≈ 46 字节
2. **在线状态统计**：`SETBIT online 1001 1`，`BITCOUNT online`
3. **布隆过滤器**：防止缓存穿透
4. **用户特征标签**：`BITOP AND/OR` 组合查询
5. **DAU 统计**：每天一个 Bitmap，`BITOP OR` 合并统计

## 三、JVM 专题

### JVM专题-01：简述一下 JVM 的内存模型

```
运行时数据区：
├─ 方法区/元空间（类信息、常量、静态变量）
├─ 堆（年轻代：Eden + Survivor0/1；老年代）
├─ 虚拟机栈（栈帧：局部变量、操作数栈、动态链接）
├─ 本地方法栈（Native 方法）
└─ 程序计数器（字节码行号指示器）
```

线程共享：堆、方法区/元空间；线程私有：栈、本地方法栈、程序计数器。

### JVM专题-02：说说堆和栈的区别

| 特性 | 堆 | 栈 |
|------|-----|-----|
| 存储内容 | 对象实例 | 局部变量、基本类型、引用 |
| 线程共享 | 共享 | 私有 |
| 分配方式 | 动态 | 编译期确定 |
| 生命周期 | GC 决定 | 方法结束销毁 |
| 速度 | 慢 | 快 |
| 溢出 | OOM | StackOverflowError |

---

### JVM专题-03：简述 Java 的对象结构

```
对象头（12 bytes，64位压缩指针）
├─ Mark Word（8 bytes）：哈希码、GC年龄、锁状态
└─ Class Pointer（4 bytes）：指向类元数据
实例数据：成员变量
对齐填充：8 字节对齐
```

---

### JVM专题-04：如何判断对象可以被回收

**可达性分析算法**：从 GC Roots 出发，不可达的对象可回收。

**GC Roots**：虚拟机栈局部变量、方法区静态属性/常量、本地方法栈 JNI 引用、同步锁持有对象、JVM 内部引用。

**回收过程**：第一次标记 → 筛选（检查 finalize）→ 第二次标记（执行 finalize，仍未关联则回收）。

---

### JVM专题-05：你知道哪些垃圾收集算法

| 算法 | 思想 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记存活，清除未标记 | 简单 | 产生碎片 |
| 标记-复制 | 存活对象复制到另一块 | 无碎片 | 内存利用率 50% |
| 标记-整理 | 存活对象向一端移动 | 无碎片 | 移动耗时 |
| 分代收集 | 年轻代复制，老年代标记-清除/整理 | 针对优化 | 复杂 |

---

### JVM专题-06：虚拟机为什么使用元空间替换了永久代

**永久代（JDK 7-）**：堆内存中，大小固定，动态生成类易 OOM。
**元空间（JDK 8+）**：本地内存，自动扩展。

**原因**：避免 OOM；统一 HotSpot 与 JRockit；灵活管理；简化 Full GC。

---

### JVM专题-07：JVM 中的即时编译器（JIT）如何工作？

**流程**：解释执行 → 热点探测（方法调用计数器/回边计数器）→ 编译执行（C1/C2）→ OSR 栈上替换。

**优化技术**：方法内联、逃逸分析、锁消除、标量替换、空值检查消除。

---

### JVM专题-08：什么是内存屏障？JVM 如何使用内存屏障保证指令执行顺序？

**内存屏障**：CPU 指令，防止指令重排序。

**四种**：LoadLoad、StoreStore、LoadStore、StoreLoad（全能，开销最大）。

**应用**：`volatile` 写前后插入 StoreStore + StoreLoad；`synchronized` 解锁插入 StoreLoad。

---

### JVM专题-09：如何排查和解决 JVM 内存泄漏问题？有哪些常用的工具和方法？

**步骤**：确认泄漏（监控堆曲线）→ 获取堆转储（`jmap -dump`）→ 分析（MAT/VisualVM/Arthas）→ 定位（大对象、GC Roots 路径）。

**常见泄漏**：静态集合持有、未关闭资源、监听器未移除、ThreadLocal 未 remove、缓存无过期。

**工具**：jstat、jmap、jstack、jconsole、VisualVM、MAT、Arthas。

---

### JVM专题-10：简述 CMS 垃圾收集器的工作流程，它有什么优缺点？

**CMS**：以最短停顿时间为目标。

**流程**：初始标记（STW）→ 并发标记 → 并发预清理 → 重新标记（STW）→ 并发清除 → 并发重置。

**优点**：并发收集，停顿短。
**缺点**：CPU 敏感；浮动垃圾；内存碎片；Concurrent Mode Failure（退化为 Serial Old）。

**JDK 14 已移除**，推荐 G1/ZGC。

---

### JVM专题-11：什么是类的初始化时机？类加载和类初始化有什么区别？

**加载过程**：加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载。

**初始化时机**：new 实例化、访问静态变量/方法（final 常量除外）、反射、子类初始化、main 方法类、接口 default 方法。

**不触发**：引用父类静态字段、定义数组、引用常量（编译期入常量池）。

---

### JVM专题-12：JVM 的栈内存中可能发生哪些异常？如何避免？

**StackOverflowError**：栈深度超限（无限递归）。避免：控制递归深度、增大 `-Xss`。

**OutOfMemoryError**：线程过多。避免：使用线程池、减少线程数。

---

### JVM专题-13：什么是直接内存？它与堆内存有何区别？如何管理直接内存？

**直接内存**：本地内存，`ByteBuffer.allocateDirect()` 分配，不受 `-Xmx` 限制。

**区别**：堆内存由 GC 管理；直接内存需手动释放（`((DirectBuffer) buffer).cleaner().clean()`），零拷贝适合 NIO。

**管理**：限制 `-XX:MaxDirectMemorySize`；重复使用（池化）。

---

### JVM专题-14：如何调优 JVM 参数以提高应用性能？请举例说明常用的调优参数及其作用

**堆内存**：`-Xms4g -Xmx4g -Xmn2g -XX:SurvivorRatio=8`

**收集器**：`-XX:+UseG1GC -XX:MaxGCPauseMillis=200`（低延迟）；`-XX:+UseZGC`（大内存）

**元空间**：`-XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=256m`

**GC 日志**：`-XX:+PrintGCDetails -Xloggc:gc.log`

**JIT**：`-XX:ReservedCodeCacheSize=256m -XX:+TieredCompilation`

---

### JVM专题-15：什么是方法区？在不同 JDK 版本中，方法区的实现有何变化？

**方法区**：JVM 规范定义的内存区域，存储类结构信息、常量、静态变量、JIT 编译代码。

| JDK | 实现 | 位置 |
|-----|------|------|
| 6- | 永久代（PermGen） | 堆内存 |
| 7 | 永久代（字符串常量池、静态变量移入堆） | 堆内存 |
| 8+ | 元空间（Metaspace） | 本地内存 |

### JVM专题-16：JVM、JRE 和 JDK 之间的关系是什么？

```
JDK（开发工具包）
  ├─ JRE（运行时环境 = JVM + 核心类库）
  │   └─ JVM（虚拟机）
  └─ 开发工具（javac、javadoc 等）
```

---

### JVM专题-17：什么是 JVM 的内存碎片？哪些垃圾收集器可以有效减少内存碎片？

**碎片**：标记-清除算法产生的不连续空闲内存。

**减少碎片的收集器**：Serial Old（标记-整理）、Parallel Old（标记-整理）、G1（Region 复制）、ZGC（并发整理）、Shenandoah（并发整理）。

---

### JVM专题-18：JVM 如何处理异常？try-catch-finally 结构在字节码层面是如何实现的？

**机制**：异常表记录 try-catch 范围和跳转地址；`athrow` 指令抛出异常，遍历栈帧查找处理器。

**finally**：字节码复制到正常路径和每个异常处理路径末尾，确保必执行。

---

### JVM专题-19：JVM 是什么？为什么需要 JVM？

**JVM**：运行 Java 字节码的抽象计算机。

**需要原因**：平台无关性（WORA）；安全性（字节码验证、沙箱）；自动内存管理（GC）；性能优化（JIT）；语言无关性（支持 Kotlin、Scala 等）。

---

### JVM专题-20：JVM 如何进行类卸载？

**条件**：该类所有实例已回收；加载该类的 ClassLoader 已回收；Class 对象无引用。

**场景**：自定义 ClassLoader（Tomcat WebAppClassLoader、OSGi 动态模块）。

**注意**：系统类加载器加载的类通常不卸载；频繁动态生成类可能导致元空间 OOM。

## 五、MySQL 专题

### Mysql专题-01：数据库的三范式是什么？

**1NF（原子性）**：每个字段值不可再分。
**2NF（完全依赖）**：非主键字段完全依赖主键（联合主键时不能只依赖一部分）。
**3NF（消除传递依赖）**：非主键字段直接依赖主键，不能传递依赖。

**反范式**：实际项目中为查询性能有意违反范式（空间换时间）。

---

### Mysql专题-02：说说 InnoDB 与 MyISAM 的区别

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | 支持 ACID | 不支持 |
| 锁粒度 | 行级+表级 | 表级 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 索引 | 聚簇索引 | 非聚簇索引 |
| 全文索引 | 5.6+ 支持 | 支持 |
| 适用场景 | 高并发事务 | 读多写少 |
| MVCC | 支持 | 不支持 |

**默认使用 InnoDB**（MySQL 5.5+），MySQL 8.0 已废弃 MyISAM 系统表。

---

### Mysql专题-03：drop、truncate、delete 的区别？

| 特性 | DROP | TRUNCATE | DELETE |
|------|------|----------|--------|
| 类型 | DDL | DDL | DML |
| 删除内容 | 表结构+数据+索引 | 所有数据（保留结构） | 指定数据 |
| 回滚 | 不可 | 不可 | 可回滚 |
| 速度 | 最快 | 快 | 慢（逐行） |
| 自增 ID | - | 重置为 1 | 不重置 |
| WHERE | 不支持 | 不支持 | 支持 |

---

### Mysql专题-04：聊一聊数据库事务机制

**ACID**：
- **原子性**：undo log 实现回滚
- **一致性**：原子性+隔离性+持久性共同保证
- **隔离性**：锁 + MVCC
- **持久性**：redo log + binlog

**实现**：BEGIN → 执行 SQL（记录 undo log）→ 修改 Buffer Pool → 记录 redo log（Prepare）→ 写 binlog → COMMIT（redo log Commit）。

**两阶段提交**：保证 redo log 和 binlog 一致性。

---

### Mysql专题-05：聊聊 MySQL 中的关联查询

**类型**：INNER JOIN、LEFT JOIN、RIGHT JOIN、CROSS JOIN。

**算法**：Nested Loop Join（小表驱动大表）、Block Nested Loop Join（join buffer）、Hash Join（MySQL 8.0+，大数据量）。

**优化**：小表驱动大表；关联字段加索引；避免 SELECT *；使用 EXPLAIN 分析。

---

### Mysql专题-06：事务隔离级别有哪些？MySQL 的默认隔离级别是什么

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现 |
|----------|------|-----------|------|------|
| READ UNCOMMITTED | 允许 | 允许 | 允许 | 无锁 |
| READ COMMITTED | 禁止 | 允许 | 允许 | MVCC（每次读生成 ReadView） |
| REPEATABLE READ（默认） | 禁止 | 禁止 | 部分禁止 | MVCC（事务开始时生成 ReadView） |
| SERIALIZABLE | 禁止 | 禁止 | 禁止 | 锁 |

**MySQL 默认**：REPEATABLE READ（通过 MVCC + 间隙锁解决幻读）。

---

### Mysql专题-07：分库分表之后，id 主键如何处理？

1. **数据库自增**：设置不同起始值和步长（如 DB1: 1,3,5... DB2: 2,4,6...）
2. **Snowflake 算法**：Twitter 分布式 ID，64 位 long 型，含时间戳、机器 ID、序列号
3. **UUID**：全局唯一，但无序、占用空间大、性能差
4. **Redis 自增**：`INCR` 生成全局唯一 ID
5. **Leaf 算法**：美团开源，号段模式（批量获取 ID 段）

---

### Mysql专题-08：说说在 MySQL 中一条查询 SQL 是如何执行的？

```
1. 连接器：建立连接，验证身份
2. 查询缓存：命中则返回（MySQL 8.0 已移除）
3. 分析器：词法分析、语法分析
4. 优化器：生成执行计划，选择索引
5. 执行器：调用存储引擎接口执行
6. 存储引擎：从磁盘/Buffer Pool 读取数据
7. 返回结果给客户端
```

---

### Mysql专题-09：讲解下 DDL DML DCL

| 类型 | 全称 | 语句 | 说明 |
|------|------|------|------|
| **DDL** | Data Definition Language | CREATE、DROP、ALTER、TRUNCATE | 定义数据库结构 |
| **DML** | Data Manipulation Language | SELECT、INSERT、UPDATE、DELETE | 操作数据 |
| **DCL** | Data Control Language | GRANT、REVOKE | 控制权限 |
| **TCL** | Transaction Control Language | COMMIT、ROLLBACK、SAVEPOINT | 事务控制 |

---

### Mysql专题-10：存储过程和触发器的作用

**存储过程**：预编译的 SQL 语句集合，封装在数据库中。
- 优点：减少网络传输、提高性能、封装业务逻辑、安全性高
- 缺点：调试困难、移植性差、增加数据库负担

**触发器**：自动执行的存储过程，由事件（INSERT/UPDATE/DELETE）触发。
- 优点：自动执行、保证数据一致性、审计日志
- 缺点：隐藏逻辑难以维护、性能影响、级联触发复杂

- `@Autowired` / `@Resource`：依赖注入
- `@Service` / `@Repository` / `@Component`：组件标记
- `@Value` / `@ConfigurationProperties`：配置绑定
- `@Transactional`：事务管理

### Springboot专题-03：简述 SpringBoot 的自动配置原理，@EnableAutoConfiguration 注解的作用是什么？

**自动配置原理**：
1. `@EnableAutoConfiguration` 通过 `@Import` 导入 `AutoConfigurationImportSelector`
2. `AutoConfigurationImportSelector` 读取 `META-INF/spring.factories`（Spring Boot 2.7+ 改为 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`）
3. 加载所有 `EnableAutoConfiguration` 对应的自动配置类
4. 通过 `@ConditionalOnXxx` 条件注解过滤，满足条件的生效

**核心设计理念**：约定优于配置，用户自定义优先（`@ConditionalOnMissingBean`）。

### Springboot专题-05：SpringBoot 的父依赖（spring-boot-starter-parent）有什么作用？如果不使用父依赖该如何配置？

**父依赖作用**：
1. 统一依赖版本管理（`dependencyManagement`）
2. 统一插件版本管理
3. 默认配置（Java 版本、编码 UTF-8、资源过滤等）
4. 打包配置（`repackage` goal）

**不使用父依赖**（公司已有父 POM）：
```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

### Springboot专题-06：如何在 SpringBoot 中读取自定义配置文件？

**方式一：@PropertySource**
```java
@Configuration
@PropertySource("classpath:custom.properties")
public class CustomConfig {
    @Value("${custom.key}")
    private String key;
}
```

**方式二：@ConfigurationProperties + @PropertySource**
```java
@Configuration
@PropertySource("classpath:custom.yml")
@ConfigurationProperties(prefix = "custom")
@Data
public class CustomProperties {
    private String key;
    private Integer value;
}
```

**方式三：Environment**
```java
@Autowired
private Environment env;
String value = env.getProperty("custom.key");
```

### Springboot专题-07：@ConfigurationProperties 和 @Value 注解的区别是什么？分别适用于什么场景？

| 特性 | @Value | @ConfigurationProperties |
|------|--------|-------------------------|
| 绑定方式 | 单个属性 | 批量绑定（对象） |
| 松散绑定 | 不支持 | 支持（`user-name` → `userName`） |
| SpEL | 支持 | 不支持 |
| 复杂类型 | 不支持 | 支持（List、Map、嵌套对象） |
| 校验 | 不支持 | 支持（`@Validated` + JSR-303） |
| 适用场景 | 简单单个属性 | 复杂配置对象 |

### Springboot专题-08：SpringBoot 启动时会扫描哪些包下的注解？如何自定义扫描包路径？

**默认扫描**：启动类所在包及其子包。

**自定义扫描**：
```java
@SpringBootApplication
@ComponentScan(basePackages = {"com.example.controller", "com.example.service"})
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

**注意**：`@ComponentScan` 会覆盖 `@SpringBootApplication` 的默认扫描，需显式包含原包。

### Springboot专题-09：SpringBoot 支持哪些日志框架？默认使用的是哪个？如何配置日志级别和输出格式？

**支持框架**：Logback（默认）、Log4j2、Java Util Logging（JUL）。

**默认**：Logback（`spring-boot-starter-logging`）。

**配置**（`application.yml`）：
```yaml
logging:
  level:
    root: INFO
    com.example: DEBUG
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n"
  file:
    name: app.log
```

**Logback 配置文件**：`logback-spring.xml`
```xml
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>
</configuration>
```

### Springboot专题-10：SpringBoot 自动配置的实现流程（从启动到自动配置生效的关键步骤）？

```
1. @SpringBootApplication
   ├─ @SpringBootConfiguration（标记配置类）
   ├─ @EnableAutoConfiguration（开启自动配置）
   │   └─ @Import(AutoConfigurationImportSelector.class)
   │       └─ SpringFactoriesLoader.loadFactoryNames()
   │           └─ 读取 META-INF/spring.factories（或 AutoConfiguration.imports）
   │               └─ 加载所有自动配置类
   └─ @ComponentScan（组件扫描）

2. 条件过滤（@ConditionalOnXxx）
   ├─ @ConditionalOnClass：类路径存在某类时生效
   ├─ @ConditionalOnMissingBean：容器中无该 Bean 时生效
   ├─ @ConditionalOnProperty：配置属性满足条件时生效
   └─ ...

3. 自动配置类生效
   └─ 注册 Bean 到 Spring 容器

4. 用户自定义优先
   └─ @ConditionalOnMissingBean 保证用户自定义 Bean 覆盖默认配置
```

**排查自动配置**：`debug: true`（打印 CONDITIONS EVALUATION REPORT）或 `/actuator/conditions` 端点。
